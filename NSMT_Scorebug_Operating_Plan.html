NSMT Scorebug Secure Live Operations Plan

Prepared for the thensmt/scorebug repo. PDF export generated from the finalized planning pass.

Summary

  • Keep the overlay public, but make it read-only and safe.
  • Make the overlay elastic by scaling a fixed 1920x1080 scene with contain, so it fits any browser window without distortion.
  • Move production hosting from GitHub Pages to Firebase Hosting with a short custom subdomain, so YoloBox URLs become short and clean.
  • Keep admin inside yolo-control.html for Phase 1, but only as a separately unlocked owner mode with its own backend-issued session.
  • Replace the current client-side PIN model with server-validated PINs, short-lived sessions, strict rules, and event-scoped state.
  • Build around eventId from the start so one game uses live now, and multiple simultaneous games later use separate isolated event routes.

Key Decisions

Overlay Fit

  • Use preserve-16:9 contain scaling.
  • Do not stretch to fill by default.
  • Keep the internal broadcast layout as a logical 1920x1080 scene.

Admin Model

  • Phase 1: no separate admin.html.
  • Owner/admin lives inside yolo-control.html as a hidden, separately authenticated mode.
  • Phase 2: add a separate owner oversight hub when running multiple simultaneous games.

Hosting

  • Use Firebase Hosting now.
  • Put a short custom subdomain in front of it, for example bug.yourdomain.com.
  • Public route model:
    • Overlay: /o/{eventId}
    • Control: /c/{eventId}

Security

  • Operator auth remains PIN-based for workflow simplicity.
  • Owner/admin also remains PIN-based in Phase 1 because that was the selected workflow.
  • Because of that choice, owner PIN auth is the weakest part of the design and must be hardened aggressively.

Implementation Changes

1. Overlay Responsiveness

  • Wrap the full overlay DOM in a top-level #scene container.
  • Change root page sizing from fixed 1920x1080 to full-window browser sizing.
  • On load and resize, compute scale = min(window.innerWidth / 1920, window.innerHeight / 1080).
  • Center the scene and scale it with CSS transform.
  • Keep existing absolute-positioned overlay internals unchanged where possible.
  • Refactor the score-pop effect so it positions relative to the scene wrapper instead of document.body.

2. Admin Inside Control

  • Split yolo-control.html into two capability layers.
  • Operator mode:
    • score
    • clock
    • period and possession
    • approved live graphics
    • active game metadata
  • Owner mode:
    • rotate operator PIN
    • reset event state from template
    • revoke active operator session
    • create and close event
    • lock and unlock control
    • copy crew URLs
  • Owner mode must require its own backend-issued owner session and never appear unless owner unlock succeeds.

3. PINs, Sessions, and Auth

  • Remove all browser-side PIN checking, direct /pin reads, and the master reset key.
  • Replace with backend validation:
    • Operator PIN: 6 digits minimum, unique per event, rotated every game, one active editor session per event.
    • Owner PIN: 10 digits minimum if numeric-only, separate validation path and session role, shorter session lifetime than operator, stricter rate limits and lockouts.
  • Store only hashed PINs, never raw PINs.
  • Backend issues short-lived Firebase-authenticated sessions after successful validation.
  • Add audit logs for unlocks, PIN rotation, session revocation, event reset, and emergency takeover.

4. Database and Rules

  • Replace the flat root model with:
    • /publicEvents/{eventId}/{bug,corners,ticker,stats,meta}
    • /adminEvents/{eventId}
    • /controlSessions/{uid}
    • /auditLogs/{eventId}/{logId}
  • Rules:
    • Public read allowed only on /publicEvents/{eventId}.
    • Public write denied everywhere.
    • Admin, session, and audit paths blocked from public clients.
    • Writes to /publicEvents/{eventId} allowed only for valid, unexpired, unrevoked sessions scoped to that exact event.

5. Assets

  • Stop storing logos and corner graphics as base64 in Realtime Database.
  • Upload assets to Cloud Storage.
  • Store only project-controlled Storage paths or signed download URLs in event state.
  • Do not allow arbitrary third-party remote image URLs for live assets.

6. Hosting and URL Cutover

  • Move production hosting from GitHub Pages to Firebase Hosting.
  • Keep GitHub as source control.
  • Use a short custom subdomain, for example:
    • https://bug.yourdomain.com/o/live
    • https://bug.yourdomain.com/c/live
  • During migration:
    • build and validate Firebase preview first
    • connect the domain
    • test live routes and event routing
    • keep GitHub Pages temporarily as rollback only during the migration window
    • retire GitHub Pages as the live host after production validation

7. Multi-Game Scale

  • Make eventId mandatory from Phase 1, even if the first event is just live.
  • Final route model:
    • overlay /o/{eventId}
    • control /c/{eventId}
  • Each event gets one overlay URL, one control URL, one operator PIN, one owner PIN, one crew label, and one game label.
  • If eventId is missing or invalid, block editing and show an error instead of falling back to shared state.

Rollout And Test Plan

  1. Secure foundation: add event-scoped data paths, backend PIN validation, short-lived sessions, RTDB rules, audit logs, and Cloud Storage assets. Remove raw /pin, client-side PIN checks, and the master key.
  2. Overlay fit: add the responsive scene wrapper and verify preserve-16:9 behavior on desktop, iPad, phone, and YoloBox.
  3. Hosting cutover: stand up Firebase Hosting preview, connect the short custom subdomain, route /o/live and /c/live, and cut over only after preview validation passes.
  4. Multi-event: add per-event routing, event badges, event creation flow, and a 5-game dry run.
Required validation: public overlay loads with no auth; public clients cannot read admin/session/audit paths; invalid PINs rate-limit and lock temporarily; revoked or expired sessions lose write access immediately; owner mode cannot be entered without a separate owner session; each event updates only its own overlay; images land in Storage, not RTDB; missing or invalid eventId blocks editing; Firebase preview route behavior matches production before cutover.

Assumptions And Defaults

  • Default overlay fit mode is contain.
  • Phase 1 owner/admin stays embedded inside control because that is the selected workflow.
  • Phase 1 owner/admin remains PIN-only because that is the selected auth model.
  • Because of that choice, these safeguards are mandatory:
    • 10-digit owner PIN minimum
    • separate hashed owner PIN and operator PIN
    • backend-only validation
    • separate rate limits and lockouts
    • short-lived owner sessions
    • audit logging
    • no client-readable secrets
  • Production hosting changes now to Firebase Hosting plus a short custom subdomain.
  • Phase 1 is secure single-game live.
  • Phase 2 adds per-event scale-out and a separate owner oversight hub for multi-game nights.
