What Codex Found (Production Code Review)

Codex performed a multi-perspective production audit simulating a senior frontend engineer, mobile specialist, Firebase architect, QA engineer, and broadcast engineer. It produced 13 findings prioritized by on-air risk.

P0 — Must Fix Before Any Live Game

Open 1. Foul/timeout indicator wiring is broken

The overlay JS calls buildDots($('fouls-away'), ...) and buildDots($('timeouts-away'), ...) but the HTML has no elements with those IDs. The DOM nodes are generic <div class="bug-indicators"></div> with no IDs. Foul and timeout dots never render.

  • Status: Not fixed. JS references 4 missing IDs: fouls-away, timeouts-away, fouls-home, timeouts-home.
  • Fix: Add id attributes to the indicator divs, or restructure to use data-* selectors.

Open 2. No in-run clock resync logic

If clockRunning===true and an incoming authoritative clock value differs from the locally-predicted time by more than a threshold, the overlay does not rebase. A second operator correcting the clock mid-run will not take effect on-air.

  • Status: Not fixed. The overlay updateClock() only acts on running→stopped or stopped→running transitions. Mid-run corrections are ignored.

Open 3. Multi-operator clock races

Two operators can simultaneously start/pause/reset the clock. No lock, lease, or server-timestamp model exists. Random final clock state during tense moments.

  • Status: Not fixed. All clock writes are non-transactional stateRef.update() calls.

Open 4. Full-size base64 images sent directly

Logo uploads write raw data URLs to Firebase with no resize or compression. A 3MB PNG becomes a multi-megabyte base64 string in the Realtime Database.

  • Status: Not fixed. Control panel shows a warning for files >500KB but does not compress or reject. Sponsor corner logos have no size check at all.

Open 5. Stats panel full rebuild on every state change

The overlay listens to the entire state/ ref. Every clock tick, score change, or color update triggers a full apply() which calls updateStats(), destroying and rebuilding the entire stats DOM.

  • Status: Not fixed. No change detection or isolated listeners.

P1 — Should Fix Before Season

Partial 6. No operator-visible error handling

FileReader, image decode, extraction, and Firebase update failures only log to console. No toast, banner, or retry UI for the operator.

  • Status: Partially addressed. Claude Code added try/catch and console.warn in the color extraction pipeline, but no visible operator feedback (toast/banner) was added anywhere.

Done 7. Harden color extraction (exColors)

Codex wanted better dominance logic, transparent/whitespace filtering, saturation weighting, and adaptive resolution.

  • Status: Largely done. Claude Code built a 3-pass extraction pipeline: (1) filtered by luminance + saturation-weighted, (2) unfiltered fallback, (3) average-all-pixels fallback. Also added iOS Safari fixes (no crossOrigin on data URLs, img.decode(), fresh canvas per call).
  • Remaining: No confidence score or operator preview of extracted colors before applying.

Open 8. Long strings can blow out overlay layout

No max-length enforcement in control inputs and no CSS text-overflow: ellipsis in overlay team name spans.

  • Status: Not fixed. Team code inputs have maxlength="4" but team name and event title have no limit. Overlay uses white-space: nowrap without truncation.

Open 9. Overlay clock does not self-stop at 0

If the controller disconnects or fails to send a stop signal, the overlay RAF loop keeps running at 0:00.

  • Status: Not fixed. The overlay clockTick() sets current = 0 but keeps calling requestAnimationFrame indefinitely.

Open 10. Touch targets too small for live ops

Score buttons and clock controls are compact. Fat-finger errors during a live game are likely.

  • Status: Not fixed. Score buttons use class .bs with padding: 5px 8px, well below the recommended 44px touch target.

P2 — Cleanups / Reliability Polish

Open 11. Dead or incomplete graphic-overlay paths

The overlay has a full showGraphic() system (TIMEOUT, FOUL, SUBSTITUTION, HALFTIME, FINAL card types) but there is no path in the control panel to trigger them.

  • Status: Not fixed. Dead code — fully built UI with no trigger.

Open 12. No validation for clock input or game state transitions

Clock input accepts any string. Period reset behavior for HALF/FINAL is not explicitly constrained.

  • Status: Not fixed. parseC() silently returns 0 for garbage input. No guard on setting period to FINAL while clock is running.

Open 13. No per-action Firebase failure feedback

All Firebase writes use fire-and-forget .update() with no .catch() or status handling.

  • Status: Not fixed. Operator has no way to know if a write failed during a transient disconnect.

Where Claude Code Disagreed with Codex

1. Color extraction severity (Codex P1 #7)

Codex rated color extraction hardening as P1 and implied the existing approach was inadequate. Claude Code had already significantly overhauled this across 3 commits (11baf0a, d88c559, 769643f) before the review was generated. The 3-pass pipeline with iOS Safari-specific fixes goes well beyond what Codex described. Claude Code treated this as substantially complete; Codex's review didn't acknowledge the work already done.

2. Base64 image size (Codex P0 #4)

Codex flagged raw base64 writes as P0. Claude Code's position: for a single-game scorebug with 2 team logos and up to 4 sponsor logos, the practical risk is low. Logos are typically small PNGs (10–80KB). The control panel already warns on files >500KB. A full compression pipeline adds complexity for a problem that rarely occurs in practice. However, Claude Code did not add compression — this is a deliberate deferral, not an oversight.

3. Multi-operator clock races (Codex P0 #3)

Codex called for a "clock owner/lease" lock or authoritative server timestamp model. Claude Code's position: NSMT's current workflow uses a single operator per game. A full locking protocol adds significant complexity for a scenario that doesn't match the current operational model. If multi-operator support becomes a real requirement, this should be designed holistically — not bolted onto the existing Firebase Realtime Database writes.

4. Stats panel rebuild performance (Codex P0 #5)

Codex rated this P0. Claude Code's position: the stats panel is hidden by default (display: none) and only shown when statsData.showPanel is true. When hidden, updateStats() returns immediately. The DOM rebuild only happens when the panel is visible, which is a small fraction of broadcast time. This is more accurately a P2 optimization, not a P0 blocker.

5. Foul/timeout indicator wiring (Codex P0 #1)

Claude Code agrees this is genuinely broken — the JS calls $('fouls-away') etc. but no elements with those IDs exist in the HTML. However, the control panel also has no UI to set foul counts or timeouts, so the feature is completely unwired end-to-end, not just missing DOM nodes. Fixing the overlay IDs alone doesn't help — the control panel needs foul/timeout buttons too.

How to Protect & Isolate These Two Files

The repo contains 11 HTML files across multiple overlay/control generations. It is easy to accidentally edit the wrong one, deploy a stale version, or confuse which overlay pairs with which control. Here's how to lock this down.

Problem: Too many overlays, unclear which is "live"

Current HTML files in the repo:

  • yolo-overlay.html — ACTIVE production overlay (YoloBox / Firebase)
  • yolo-control.html — ACTIVE production control panel (YoloBox / Firebase)
  • nsmt_overlay.html — Legacy overlay (pre-YoloBox)
  • nsmt_fox_overlay_ws.html — Legacy Fox-style overlay (WebSocket)
  • nsmt_fox_control_ws.html — Legacy Fox-style control (WebSocket)
  • nsmt-producer-v2.html — Legacy producer dashboard
  • stats_v3.html — Stats overlay
  • stats_control.html — Stats control
  • ticker2.html — Ticker overlay
  • index.html — Repo landing page

Solution 1: Move production files to a dedicated folder

Create a /live/ directory. Only the two active files live there:

  • /live/yolo-overlay.html
  • /live/yolo-control.html

GitHub Pages would serve them at thensmt.github.io/scorebug/live/yolo-overlay.html. Bookmark these URLs on every device. The rest of the repo stays at root for reference.

Solution 2: Add a CODEOWNERS file

Create .github/CODEOWNERS to require review for changes to the production files:

/live/yolo-overlay.html @thensmt
/live/yolo-control.html @thensmt

This prevents accidental merges that touch the live files without explicit approval.

Solution 3: Pin with a version comment

Add a version header comment to both files so you always know which build is deployed:

<!-- NSMT YoloBox Overlay v1.0 | 2026-04-03 | DO NOT EDIT without checking yolo-control.html -->

The overlay and control are a matched pair. The comment makes this explicit so nobody edits one without considering the other.

Solution 4: Archive legacy files

Move all non-active files into an /archive/ folder. This makes the repo root clean and obvious:

  • /live/ — The two production files
  • /archive/ — All legacy overlays, controls, WebSocket versions
  • / — index.html, README, config, assets

Solution 5: Rename for clarity

If you don't want to reorganize folders, rename the active files to make them unmistakable:

  • LIVE-overlay.html
  • LIVE-control.html

Anything prefixed with LIVE- is production. Everything else is legacy or experimental.
