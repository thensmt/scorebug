Production Code Review — Must-Fix Checklist
Files: yolo-overlay.html, yolo-control.html  |  Repo: thensmt/scorebug  |  April 3, 2026
Prioritized by on-air risk (impact × likelihood). This checklist covers all findings from a multi-perspective production audit simulating a senior frontend engineer, mobile specialist, Firebase architect, QA engineer, and broadcast engineer.

P0 — Must Fix Before Any Live Game

1. Fix foul/timeout indicator wiring

Add missing foul/timeout DOM nodes (with stable IDs), or remove dead reads. Ensure overlay reads/writes use the same keys and actually render.

  • Why P0: Currently silently broken feature during live game context.
  • Refs: Missing elements vs. calls in overlay apply() function.

2. Add in-run clock resync logic (overlay + control)

If clockRunning===true and incoming authoritative clock differs from local expected by threshold (e.g., >250ms or >1s), rebase local timer.

  • Why P0: Current multi-operator corrections don't apply reliably on-air.
  • Refs: Overlay/control run-state logic gaps.

3. Prevent multi-operator clock races

Introduce “clock owner/lease” lock (or authoritative server timestamp model). At minimum, prevent simultaneous start/pause/reset from non-owner.

  • Why P0: Random final clock state during tense moments.
  • Refs: Non-transactional writes for clock controls.

4. Stop sending full-size base64 images directly

Resize/compress client-side before write (e.g., max dimension + JPEG/WebP quality). Enforce max byte limit and type validation.

  • Why P0: Memory/bandwidth spikes and mobile instability.
  • Refs: Current raw data URL writes.

5. Stop stats panel full rebuild on unrelated updates

Only rerender stats when stats payload changes. Keep clock updates isolated from stats DOM reconstruction.

  • Why P0: Performance degradation over long broadcasts.
  • Refs: Full-state listener + stats rebuild loop.

P1 — Should Fix Before Season

6. Add robust operator-visible error handling

FileReader/image decode/extraction/update failures should display toast/banner + retry option.

  • Why P1: Silent failures are dangerous in live ops.
  • Refs: Console-only warnings and missing error hooks.

7. Harden color extraction (exColors)

Better dominance logic (ignore transparent whitespace more aggressively, preserve saturation, sample adaptive resolution). Keep fallback path but surface confidence/preview to operator.

  • Why P1: Recurring pain point + branding quality risk.
  • Refs: Current quantization/filtering approach.

8. Clamp and truncate long strings in overlay UI

Apply max lengths in control and CSS ellipsis/max-width in overlay (team, event fields).

  • Why P1: Prevents layout blowouts.
  • Refs: nowrap without truncation.

9. Ensure overlay clock self-stops at 0 if no controller heartbeat

Add local stop condition when countdown reaches zero and no new authoritative updates.

  • Why P1: Avoids unnecessary perpetual RAF at 0.
  • Refs: Clock loop behavior.

10. Improve touch ergonomics

Increase key controls to ~44px target height, especially score and start/pause/reset.

  • Why P1: Reduces fat-finger mis-ops.
  • Refs: Compact button sizing/layout.

P2 — Cleanups / Reliability Polish

11. Resolve dead or incomplete graphic-overlay paths

Wire to state or remove dead code to reduce confusion.

  • Why P2: Lowers maintenance and “it should work” surprises.
  • Refs: Graphic functions not triggered by listener path.

12. Add explicit validation for clock input and game states

Reject malformed times, constrain period-reset behavior for HALF/FINAL explicitly.

  • Why P2: Prevents accidental odd states.
  • Refs: Permissive parser/reset behavior.

13. Per-action Firebase failure feedback

Attach .catch/status handling to writes; show operator action result.

  • Why P2: Avoids false confidence during transient disconnects.
  • Refs: patch/update usage patterns.

Safe Execution Order

  1. Data model / clock authority changes (P0 #2, #3)
  2. Performance isolation (P0 #5)
  3. Upload pipeline limits + compression (P0 #4)
  4. UI correctness fixes (P0 #1, P1 #8, P1 #10)
  5. Error handling + extraction hardening (P1 #6, #7)
  6. Cleanup / polish (P2 items)

Minimal Acceptance Tests Before Going On-Air

Setup: Two iPhones + one overlay instance

  • Simultaneous start/pause/reset stress for 2 minutes
  • Mid-run clock correction from second device
  • 5MB image upload rejection/compression path
  • 30s offline/reconnect while clock running
  • 6+ hours soak test with periodic updates
  • Max-length team/event text and max slider values
Pass Criteria: No clock divergence, no visible stutter, no silent failures, no layout overflow.
